# あなたの Tribune を作るには
## ─ Claude Code で「自分のための朝刊」を構築するガイド


## このドキュメントについて

このガイドは、合宿メンバー（Claude Code を触ったことがある、けど非技術者）が、
自分専用の朝刊「Tribune」を構築できるようにするための実用ドキュメントです。

書き手の神山自身、Claude Code を使い始めて最初に作ったまともなプロダクトが
この Tribune でした。技術的に難しいことはほとんどしていません。難しかったのは、
むしろ「自分にとって読みたい朝刊とは何か」を言語化する側でした。

このガイドはその過程をなぞる形で書いています。Tribune を「動くもの」として
作るのは Claude Code の仕事、Tribune を「読みたいもの」として設計するのは
あなたの仕事。役割分担を最初に明確にしておくと、開発がスムーズに進みます！

**想定読者**

- Claude Code を触ったことがある（基本操作がわかる）
- プログラマーではない（本職は別、コードはほぼ Claude Code に任せる）
- 自分の関心領域や読みたいものが、ある程度言語化できている（or したい）
- 1 週間の集中、または分散して数週間（毎日 1-2 時間）の余裕がある

**完成までの所要時間の目安**

神山の場合は、合宿の半日 + GW 連休 4 日 + Sprint 5 の 2 日、合計 約 1 週間で
クラウド自動化まで到達しました。最小実装（1 面だけ）なら 1-2 日で動きます。

**このガイドで作れるもの**

- 毎朝 5:00 JST に GitHub Actions が起動 → archive HTML 生成 → Vercel デプロイ
- 約 90 の RSS ソースから美意識軸でフィルタした 6 面構成の新聞風 HTML
- ブックマークするだけで、毎朝最新版が読める運用

# 第1部：思想 ─ Tribune は何か


## 第1章 なぜ「編集された朝刊」というメタファーなのか

Tribune を作るときに最初に決めたのは、「これは何か」の輪郭でした。
RSS リーダーでもニュースアグリゲータでも対話型 AI でもない。新聞です。
しかも「編集された」新聞です。

普通のニュースサービスは、記事を時系列に並べるだけだったり、
「今日のトレンド」みたいな大衆向けの軸で並び替えるだけだったりします。
それだと「自分にとって今日読むべきもの」は浮かび上がってこない。

新聞は違います。編集者が「今日のトップはこれ」と判断して、紙面構成を組んで、
リード文を書いて、解説をつけて、見出しを工夫して、読者に届ける。
紙面そのものが一つの編集物です。

Tribune が目指したのはこれです。AI に「編集者」をやってもらう。
記事を選ぶだけじゃなく、「なぜ今朝の紙面トップなのか」をリード文で示し、
紙面全体を俯瞰した編集後記まで書く。読者（あなた自身）が手に取って
「読みごたえがある」と感じる紙面を毎朝届ける。

このメタファーは強力で、開発中の細かい判断（どんな記事を採用する／しないか、
voice はどうするか、紙面構成はどう組むか）すべてに効きました。
「新聞らしいか？」が判定基準になるからです。

あなたの Tribune を作る前に、まず「自分にとっての新聞とは何か」を
1-2 文で言語化してみてください。「編集された朝刊」に共感するか、
別のメタファー（雑誌？日記？対話相手？）の方がしっくりくるか。
ここがブレると、その後の判断もブレます。

→ 次章では、Tribune の核心である「美意識軸」について書きます。


## 第2章 美意識軸を持つことの意味

Tribune の中核には「美意識軸」という考え方があります。
これは「自分にとって良い記事の条件」を、複数の独立した軸として
言語化したものです。神山さんは 7 軸を設計しています。

たとえば「規模より本質」という軸があります。これは「100億円調達」
「業界初」みたいな数字や派手さで語る記事より、現象の構造や因果を
丁寧に書いた記事を優先する、という判断軸です。

別の軸として「主流より目利き」があります。Smart News に載るような
バイラル記事より、専門家しか読まないような niche なメディアの方を
優先する。日経のヘッドラインより、Foresight や Project Syndicate の
解説の方が好み、みたいな趣味です。

ここで大事なのは、**「軸の設計」と「コードへの実装」は別物**だという
ことです。神山さんは 7 軸を設計していますが、現状の実装に明示的なスコア
として降りているのは美意識2（主流より目利き、mainstream 判定 + Foresight
ペナルティ）と美意識4（規模より本質、blacklist）の **2 軸だけ**です。
残り 5 軸は LLM プロンプト側に判断材料として伝えていて、明示的なスコア
計算式にはなっていません。

たとえば美意識4 は `config/blacklist_keywords.md` に「成功談」「カリスマ
礼賛」「金額誇示」系のキーワードを並べて減点（神山の場合 33 パターン）。
美意識2 は `sources/` 配下のメディア定義に mainstream レベルを書いて、
大手メディアの記事に減点。スコアロジック本体は `scripts/selector/`
配下の `stage1.py` などにあります。

軸を持つ意味は、「なんとなく好き／嫌い」を運用可能なルールに変えること
です。コードに落としてしまえば、毎朝同じ基準で記事を評価できる。
あなたの Tribune を作るとき、最初にやるべきは「自分の美意識軸を 5-7 個
書き出す」ことです（付録 A にワークシートがあります）。

注意点：**最初から 7 軸全部をスコア計算に降ろす必要はありません**。
神山さんも 7 軸を設計しただけで、実装に降りているのは 2 軸。
あなたの場合も、まず 5-7 軸を言語化して、優先度の高い 2-3 軸から
スコア計算に落とせば十分動きます。観察 → 修正 → 観察 のサイクルで、
必要に応じて軸を増やしていけば OK です！

→ 次章では、軸の中でも特殊な「セレンディピティ」について書きます。


## 第3章 セレンディピティの設計

「美意識軸で選ぶ」だけだと、Tribune は読者の関心領域に最適化されすぎて、
逆に窮屈になります。毎日同じような話題ばかり、自分の知ってる世界が
ループする紙面になってしまう。

これを避けるために Tribune には「セレンディピティ枠」があります。
神山版では第 5 面の上半分。普段読まない領域（カルチャー、人類学、書評、
音楽、アウトドア、料理など）から毎朝 1 記事だけランダムに選ぶ仕組みです。

ポイントは「過去 30 日に表示してないジャンル」を優先すること。
昨日「音楽」を出したなら、今日は「料理」や「書評」が優先される。
これで「同じような話題ばかり」を構造的に防げます。
実装は `scripts/page5/serendipity_selector.py` と
`scripts/selector/dedup_filter.py` に入っています。

なぜこの設計が効くかというと、「読みたいものだけ届ける」と
「読むべきものを届ける」のバランスを、構造でコントロールできるからです。
紙面の 9 割は美意識軸で最適化、1 割はランダム性、みたいな比率です。

あなたの Tribune を設計するとき、「自分が普段絶対読まないジャンル」を
3-5 個挙げてみてください。神山の場合は人類学、書評、音楽フェス情報、
ハイカー向けギア記事、料理レシピなどでした。これらが第 5 面で
予期しない出会いを毎朝もたらしてくれます。

→ 第2部では、実装の流れをフェーズごとに見ていきます。

# 第2部：実装の流れ ─ フェーズで進める


## 第4章 フェーズ1：最小実装（1 面だけの新聞）

最初のフェーズは「1 面だけの新聞を動かす」ことです。完璧を目指さない。
むしろ「動く一番小さいもの」を 1-2 日で出すのがコツ。

最小スコープの例：

- 第 1 面（トップ記事 1 本 + サイドバー）だけ
- ソースは 3-5 個（BBC News、The Economist、自分の好きなブログ等）
- RSS で取得 → スコアリング → 1 本選ぶ → HTML として出力
- ブラウザで開いて読める

これで「Tribune っぽいもの」が形になります。Claude Code に
`share/03_starter_prompt.md` を投げて対話を始めると、Sprint 1 として
これが組み上がります。

技術的には、この時点で出てくる主な要素は：

- Python（Anthropic SDK だけ依存、他は標準ライブラリ）
- RSS パース（feedparser ではなく標準 `urllib` + `xml.etree`）
- Anthropic API でスコアリング（記事を読んで美意識軸で点数化）
- HTML テンプレート（`layout/template.html` のような感じ）

神山の場合は、合宿（半日）でダミーデータを使った 1 面 HTML レイアウトが
動きました。実 RSS パイプライン（取得 → スコアリング → 出力）はその後の
GW 連休で組みました。スコアリングは雑、紙面は質素、でも「自分の関心軸で
フィルタした朝刊」が形になっている。これが大事。

ここで止めずに次へ進むコツは、「動いているうちに次の不満を見つける」こと。
最小実装を毎日数日眺めていると、必ず「ソースが足りない」「リード文が薄い」
「煽り記事が混じる」といった違和感が出てきます。それを次フェーズで潰します。

→ 次章では、ソースを拡充して選定ロジックを育てるフェーズを書きます。


## 第5章 フェーズ2：ソースの拡充と選定ロジック

最小実装が動いたら、次はソースを増やして選定ロジックを育てます。
神山の Tribune では、現在 9 カテゴリ × 約 10 ソース = 約 90 個の
RSS ソースが登録されています（`sources/` 配下、business.md / academic.md /
geopolitics.md / culture.md / books.md / music.md / outdoor.md /
cooking.md / companies.md）。

ソース拡充は「興味のある専門家／メディアの RSS をかき集める」作業です。
Claude Code に「The Economist の RSS フィード URL を教えて」と聞けば
教えてくれます。フィード URL を見つけたら `sources/<category>.md` に
登録するだけ。

選定ロジックは段階的に育てていきます：

**Stage 1（Mainstream + Blacklist）**：機械的フィルタ。
「主流より目利き」軸で大手メディアの記事に減点、「規模より本質」軸で
NG キーワード（「成功談」「カリスマ礼賛」「金額誇示」など）に減点。
ここは LLM を使わない、軽量な前処理。

**Stage 2（LLM スコアリング）**：Anthropic API で各記事を読んで、
複数の美意識軸（神山さんの場合は 7 軸を設計、ただし第 2 章で書いた通り
明示的なスコアに降りているのは美意識2 と美意識4 の 2 つ、残り 5 軸は
LLM プロンプトに判断材料として渡しています）で点数化。これが Tribune の
核心ロジックで、いちばんプロンプト設計に時間がかかる箇所です。

**Stage 3（統合スコア + 選定）**：Stage 1 と Stage 2 のスコアを統合して、
紙面ごとにトップを選ぶ。

このフェーズでは「観察 → 修正」のサイクルを毎日回します。
朝、生成された archive を読んで、違和感のある記事を探して、
プロンプトや軸を調整して、翌朝また確認。地味だけど、ここで紙面の
質感が決まります。

神山の場合、ある時期から Foresight（新潮社）が連日トップに来るように
なって、紙面の見た目が偏ってしまったことがありました。対処として
`scripts/regen_front_page_v2.py` に Foresight だけスコア -10 する
ペナルティを追加しました（`FORESIGHT_PENALTY = -10.0`）。
このように「観察して気づいた偏り」を局所修正で潰していきます。

→ 次章では、6 面構成への拡張を扱います。


## 第6章 フェーズ3：紙面構成の拡張

1 面が安定したら、6 面構成（神山版）に拡張します。各面の役割を
分けることで、紙面に「リズム」が生まれます。

神山版の構成：

- **第 1 面**：国際ニュースのトップ + サイドバー解説
- **第 2 面**：自社グループ業界ニュース（神山の場合は 3 社）
- **第 3 面**：6 領域（地政学／経済／規制／経営／テック／自然科学）から各 1 本
- **第 4 面**：今週の概念（哲学／思想／学術）+ 学術記事
- **第 5 面**：セレンディピティ枠（前章）+ 個人コラム
- **第 6 面**：余暇（読書／音楽／アウトドア／料理）

各面のコードは `scripts/page1/`〜`scripts/page6/`（または
`scripts/selector/page2.py` `page3.py`）に分かれていて、それぞれ
独立した選定ロジックを持っています。

設計のコツは「各面で何をしたいか」を先に言語化すること。たとえば
「第 2 面は自分の仕事に直接関係するニュース、ただし PR 記事は除外」
「第 4 面は週単位で 1 つの哲学的概念を取り上げて、それに関連する
学術記事を 1 本」みたいに。役割が明確だと、Claude Code が実装を
迷わずに進められます。

あなたの Tribune は、6 面である必要はありません。3 面で十分かもしれない、
逆に 8 面欲しいかもしれない。神山版を雛形に、自分の関心構造に合わせて
組み替えてください。付録 B に紙面構成テンプレートがあります。

→ 次章では、紙面に「voice」を持たせる編集要素を扱います。


## 第7章 フェーズ4：編集要素の追加（リード文・編集後記）

紙面構成が整ったら、ここから「voice」をつけていきます。
これが Tribune を単なる記事リストから「新聞」に格上げする工程です。

主な編集要素：

**リード文**：第 1 面トップ記事の冒頭 60-100 字。
記事の核心を凝縮した導入文で、毎朝 LLM が生成します。
「なぜこの記事が今朝の紙面トップなのか」が一瞬で掴める設計。
実装は `scripts/selector/why_important.py`。

**編集後記**：紙面全体を俯瞰した 100 字程度のコメント。
神山版では「Tribune 編集部」名義で書かれていて、英国紙風の
観察的な voice を意識しています。実装は `scripts/editorial/`。

**個人コラム**：第 5 面の AI かみやまコラム。神山の場合は、
自分のブログや社内発信資料を RAG として構築した分身 AI に
書かせています。聞き上手で哲学的な対話相手という人格設定。

ここで重要なのは、**voice を意図的に使い分ける**ことです。
神山版では：

- 「AI かみやま」＝ 個人の声、第 5 面コラム
- 「Tribune 編集部」＝ 媒体の声、第 1 面リード文と編集後記

この二層構造が紙面に厚みをもたらします。リード文と編集後記が
同じ voice だと一本調子になるし、コラムだけ別人格だと浮いてしまう。
**「個人の声と媒体の声が同居する」という構造**を最初に決めておくと、
Sprint 4 以降のプロンプト設計が楽になります。

あなたの Tribune では、voice を 1 種類にするのもアリです。
ただし「誰が書いているか」を曖昧にすると紙面が散漫になるので、
「これは誰の声で書かれた紙面か」を最初に決めておくのが安全です。

→ 次章では、仕上げの細部調整について書きます。


## 第8章 フェーズ5：仕上げと細部調整

ここからは「美しい紙面」を作る工程です。技術的に難しいことは
ほぼなくて、地味な観察と修正の積み重ねです。

調整対象の例：

- 翻訳ポリシー（英語ソースをどこまで日本語化するか）
- リンクのスタイル（ドット下線 + ホバー実線とか）
- 紙面のレイアウト（多段組み、ドロップキャップ、罫線）
- フォント（Playfair Display、Merriweather、Noto Serif JP）
- masthead（紙名、日付、天気、株価、為替、カウントダウン）
- 編集後記の位置と分離ライン
- スマホ対応（神山版はまだ未対応）

神山の場合、Sprint 5 の終盤からこの「細部調整フェーズ」に入って、
連日小さな修正を入れています。第 2 面のロゴ位置、編集後記の分離ライン、
リンクスタイル、リード文の長さ調整……といった調整。
1 つ 1 つは些細だけど、紙面全体の格調を作るのはこの積み重ねです。

このフェーズでのコツは、「毎朝の archive を眺めて、違和感を 1 つ拾う」
習慣をつけること。違和感はメモして、Claude Code に対話形式で投げる。
たとえば「第 2 面のロゴが社名の左にあって違和感がある、上にセンター
配置にしたい」みたいに具体的に。

技術的には、`archive/YYYY-MM-DD.html` を直接ブラウザで開いて確認して、
気になる箇所の CSS や生成ロジックを Claude Code に直してもらう、という
ワークフローです。`scripts/regen_front_page_v2.py` でテンプレートと
スタイルが管理されています。

このフェーズに「終わり」はありません。紙面は毎朝生成されるので、
毎朝改善の余地が出てくる。長期運用していくものです！

→ 次章では、自動化（クラウド化）について書きます。


## 第9章 自動化（クラウド化：GitHub Actions + Vercel）

ローカル PC で「毎朝 5 時に cron 実行」も可能ですが、PC を起動して
おく必要があり、現実的ではありません。クラウド化が解です。

神山版の構成：

- **GitHub Actions**：毎朝 UTC 20:00（= JST 5:00）に自動起動。
  ubuntu-latest VM で Python 環境を整え、Anthropic API キーを
  Secrets から注入して、`python -m scripts.regen_front_page_v2 --date $(...)`
  を実行。生成された `archive/<date>.html` と `index.html` と
  state JSON を main ブランチに自動コミット & push。
- **Vercel**：main への push を webhook で検出して自動 deploy。
  静的ホスティング、URL は `https://kt-paper-1caa.vercel.app/`。
  数十秒でデプロイ完了。

これで PC を起動する必要なし、完全クラウド自走です。

実装は `.github/workflows/daily.yml` 1 ファイルにまとまっています。
神山の場合、初期セットアップ（GHA + Vercel）は半日でしたが、
runtime に必要なファイルが gitignored だったり timeout が足りなかったり
push が競合したりで、運用に乗せるまでの微修正が積み上がって、Sprint 5
全体で約 2 日かかりました（具体的な落とし穴は第 11 章で共有します）。
詳しい手順は `SETUP.md` に書いてあります。

注意点：

- **GitHub Secrets** に `ANTHROPIC_API_KEY` などを登録する必要あり
- **timeout-minutes** は実測 + 余裕で設定（神山版は 60 分）
- **logs/state JSON** は git に commit して翌日の dedup state を維持
- **`docs/` は gitignored** なので、runtime に必要な config は
  `config/` に置くこと（神山は最初これでハマりました、第 11 章で詳述）

GitHub Actions の無料枠は private repo で月 2,000 分。60 分 × 30 日
= 1,800 分なので余裕で収まります。Vercel も Hobby プランで無料。
**ランニングコスト 0 円、API 利用料だけ**で運用できます。

→ 第 3 部では、Claude Code との付き合い方について書きます。

# 第3部：Claude Code との付き合い方


## 第10章 設計判断は人間が握る

Tribune を作る上で最も大事な原則は、**設計判断は人間が握る**ことです。
Claude Code は実装の達人ですが、「Tribune とは何か」を決めるのは
あなたの仕事です。

具体的に「人間が決めるべきこと」と「Claude Code に任せて良いこと」
の例を挙げると：

**人間（あなた）が決めること**

- 紙面構成（何面構成にするか、各面の役割）
- 美意識軸（記事を選ぶ判定基準）
- voice（誰の声で書くか、何種類使い分けるか）
- ソース選定（どのメディアを採用／除外するか）
- 個別の編集判断（このペナルティは妥当か、この記事は OK か）

**Claude Code に任せて良いこと**

- Python のコード実装（モジュール分割、関数設計、エラーハンドリング）
- HTML/CSS の整形（レイアウト、フォント指定、罫線）
- テストの書き方
- GitHub Actions / Vercel のセットアップ
- ライブラリの選定（PyYAML を使うか、標準 ライブラリで済ますか等）

判断の境界線は「これが Tribune の質感に影響するか？」で線引きします。
影響するなら人間、影響しないなら Claude Code。

`share/03_starter_prompt.md` でも触れていますが、最初のメッセージで
**「設計判断は私が握ります」「実装はあなたに任せます」と明示する**と、
Claude Code は健全に動いてくれます。明示しないと、Claude Code は
気を利かせて勝手に設計判断もやりがちで、後で「あれ、こう決めたつもりじゃ
なかったのに」となります。

神山の場合、Sprint 3 までは Claude Code が暴走気味で、頻繁に
「ちょっと待って、それは私が決める」と差し戻していました。
**最初に役割分担を明確にすればよかった**、というのが Sprint 4
以降の学びです。

→ 次章では、神山が経験した失敗例を共有します。


## 第11章 失敗から学んだこと

ここまで滑らかに書いてきましたが、実際の開発はトラブルの連続でした。
いくつか印象的な失敗を紹介します。同じ罠を踏まないように！

**37 分の承認待ち事件**

ある朝、フル archive 再生成を走らせたら 41 分かかりました。
通常 8-10 分で終わるはず。原因を追ったら、Claude Code が承認ダイアログを
表示したまま 37 分待っていただけ、実処理は 4 分でした。

教訓：**Claude Code の権限設定（`.claude/settings.json`）は最初に整える**。
頻繁に使うコマンドは allow list に登録、危険なコマンドは deny に。
神山の場合は `Bash(python3 -m scripts.regen_front_page_v2:*)` を
プロジェクト共有 settings に登録して解決しました。

**focus_work が空文字列で発動しなかった事件**

第 6 面の趣味記事に「今朝の焦点は○○」のラベルを表示する仕様を
入れたつもりが、空文字列で発動条件が不発、ラベルが出ませんでした。
LLM 生成の戻り値が空のときの fallback 経路が足りなかった。

教訓：**LLM の出力が空／不正な場合の fallback 経路を必ず書く**。
LLM は気まぐれです。「動く前提」じゃなく「壊れた時にどう振る舞うか」
を最初に設計しておく。

**GHA で FileNotFoundError 連発（×2 回）**

クラウド化したら、初回のラン #1 で `docs/blacklist_v1.md` がない
というエラー。`docs/` は個人 design docs として gitignored だったので、
GitHub Actions の runner には存在しなかった。修正してラン #2 を投入したら、
今度は `docs/companies_context_v1.md` で同じエラー。

教訓：**runtime に必要な設定ファイルは `config/` に置いて track**。
`docs/` は「個人ノート」、`config/` は「実行時 config」、と
ディレクトリの役割分担を最初に決めておく。

**GHA timeout 30 分で archive 生成途中で cancel された事件**

ローカルでの dry-run が 23 分だったので、timeout を 30 分に設定。
本番ラン #3 は 30 分 11 秒かかって、11 秒 over で強制 cancel されました。
archive は途中まで生成されたが push されず、消滅。

教訓：**timeout は実測 + 50% 以上の余裕で設定する**。
最終的に 60 分にしました。GitHub Actions の無料枠（月 2,000 分）に
対しては誤差です、ケチるところじゃなかった。

**ラン #4 で push が競合して archive が消滅した事件**

ラン #4 が走っている最中（30 分間）に、別作業で `share/` に
ドキュメントを push したら、ラン #4 の最後で `git push` が
non-fast-forward で reject。archive は runner 上で生成されたが
push されず、また消滅。

教訓：**workflow の `git push` の前に `git pull --rebase` を入れる**。
並行 push があっても安全に取り込める。これは最初から入れておけば
よかった、典型的な落とし穴です。

これらの経験から学んだメタな教訓は、**「クラウド化したら、ローカル
で動いていたコードが想定外の理由で壊れる」**ということ。ローカル開発
の前提（gitignored ファイルの存在、PC 内の永続状態、ユーザー操作との
時間的隔たり）が、クラウドでは成り立たないことが多い。
クラウド化フェーズは「想定外を潰す」工程と思って臨んでください！

→ 次章では、これから始めるあなたへ、最後のメッセージです。


## 第12章 これから始めるあなたへ

ここまで読んでくれてありがとうございます。

最後にいくつか、これから始めるあなたへの言葉を：

**最小実装を急いで作る**：完璧な設計を先に固めようとしないでください。
1 面だけの新聞でいいから、まず動かす。動いてから違和感を見つける。
神山の場合は合宿の半日でダミーデータの 1 面レイアウトが動き、
GW 連休で実 RSS パイプを組みました。

**毎朝 archive を眺める習慣**：これが Tribune を育てる最大のコツです。
朝コーヒーを飲みながら自分の Tribune を読む。違和感をメモする。
Claude Code に投げる。翌朝確認する。このサイクルを数週間回すと、
紙面が自分の感性に合うように仕上がります。

**Claude Code との対話を楽しむ**：これは「ツールを使う」というより
「相棒と仕事する」感覚に近いです。設計判断はあなたが握りつつも、
実装の細部や代替案について Claude Code と相談すると、思いがけず
良いアイデアが出てきます。

**完成した Tribune を誰かに見せる**：神山の場合、合宿メンバーに
見せることがモチベーションになりました。誰かに見せる前提だと、
紙面の細部まで気を遣うようになる。今これを読んでいるあなたが
神山にとっての「誰か」です、ぜひ自分の Tribune も見せてください。

**完成しないことを恐れない**：Tribune は「育てるメディア」です。
v1.0 で完成、ではなくて、毎朝少しずつ育つもの。今日完璧じゃなくて
いい、明日少し良くする。それで OK です。

詰まったらこのチャットで聞いてください。あなたの Tribune が
完成する日を楽しみにしています！

# 付録


## 付録A 美意識軸ワークシート

自分の美意識軸を 5-7 個書き出してみましょう。各軸について、
「好きな例」と「嫌いな例」をペアで書くと言語化しやすいです。

**書き方の例**（神山版から 1 つ）：

> **軸名**: 規模より本質
>
> **意味**: 数字や派手さで語る記事より、現象の構造や因果を丁寧に
> 書いた記事を優先する。
>
> **好きな例**: McKinsey の構造分析、Foresight の地政学解説
>
> **嫌いな例**: 「100億円調達」「業界初」「○年連続増収」のような
> 数字／規模を強調する見出し

**あなたの番です**：

```
軸 1
名前:
意味:
好きな例:
嫌いな例:

軸 2
名前:
...
```

5-7 個書き出せたら、Claude Code に渡してください。Tribune の
スコアリングロジックに反映されます！


## 付録B 紙面構成テンプレート

紙面構成の例。神山版は 6 面ですが、3 面で十分かもしれないし、
8 面欲しいかもしれません。

**最小構成（3 面）**

- 第 1 面：トップ記事 1 本（あなたの最重要関心領域）
- 第 2 面：セカンダリ 3 本（関心領域の周辺）
- 第 3 面：余暇 1 本（仕事と関係ない発見）

**中規模構成（5 面）**

- 第 1 面：国際 / 主要トピック
- 第 2 面：仕事関連（業界 / 自社グループ）
- 第 3 面：関心領域別（3-5 領域）
- 第 4 面：学術 / 思想
- 第 5 面：セレンディピティ + 余暇

**フル構成（神山版、6 面）**

序章で見た構成。第 6 面まで使って余暇を独立させる。

各面に「役割」を 1 文で書けるか試してください。書けないなら、
その面はまだ要らないかもしれません。

## 付録F よくあるエラーと対処

**`FileNotFoundError: docs/<file>.md`**

→ `docs/` が gitignored で GHA runner に存在しない。
`config/` に移動して track。第 11 章参照。

**GHA timeout で archive 生成途中で cancel**

→ `timeout-minutes` を増やす（神山版は 60 分）。
実測 + 50% 以上の余裕を取る。

**GHA の `git push` が non-fast-forward で reject**

→ workflow の `git push` の前に `git pull --rebase origin main` を
入れる。並行 push 対応。

**`ANTHROPIC_API_KEY: not set` などのエラー**

→ GitHub Secrets に登録忘れ。Settings → Secrets and variables → Actions
で `ANTHROPIC_API_KEY` を登録。

**Claude Code の承認ダイアログで実行が止まる**

→ `.claude/settings.json` の allow list に頻用コマンドを登録。
プロジェクト共有 settings.json なら git で track 可能。第 11 章参照。

**`DAILY_COST_CAP_USD exceeded` で abort**

→ LLM コスト上限に達した。`scripts/regen_front_page_v2.py` で
`DAILY_COST_CAP_USD` を確認。Sprint 中の引き上げが必要なら
一時的に上げる、運用安定後は元に戻す。

詰まったらこのチャットで聞いてください！
